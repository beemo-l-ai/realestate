# Development Rules

## 1) 공통 코딩 규칙

- TypeScript `strict` 기준을 유지합니다.
- 외부 입력(API/툴 인자/환경변수)은 항상 검증합니다.
- 숫자/날짜/통화는 저장 형식을 명확히 고정합니다(예: `YYYYMM`, KRW 정수).
- 한 함수가 두 가지 이상의 책임을 가지지 않도록 분리합니다.
- 사이드이펙트(저장/네트워크) 함수와 순수 계산 함수를 분리합니다.

## 2) MCP 서버 규칙 (`src/mcp`)

- 툴 이름은 동사+도메인 형태로 명확히 정의합니다.
- 툴 입력은 `zod` 스키마로 강제합니다.
- 응답은 `content`(사람 읽기) + `structuredContent`(기계 처리)를 함께 제공합니다.
- "수도권 한정" 범위를 벗어나는 질의에는 제한을 명시합니다.

## 3) 수집기 규칙 (`src/collector`)

- 원본 필드명과 정규화 필드명을 분리해 유지합니다.
- 거래 ID는 재수집 시 동일하게 생성되는 결정적 규칙을 유지합니다.
- API 실패는 상태코드/문맥을 포함한 에러로 반환합니다.

## 4) 저장소 규칙 (`src/lib`)

- Firestore 컬렉션명은 상수로 일원화합니다.
- 집계 로직은 순수 함수로 유지하여 테스트 가능하게 만듭니다.
- 문서 ID 규칙 변경 시 반드시 마이그레이션 계획을 기록합니다.

## 5) 스크립트 규칙 (`src/scripts`)

- CLI 인자 기본값을 제공하고, 잘못된 인자는 즉시 실패시킵니다.
- 배치 로그는 지역/월/건수 등 핵심 운영 지표를 포함합니다.
- 재실행 시 중복 삽입이 발생하지 않도록 upsert 경로를 사용합니다.

## 6) 문서/구조 규칙

- 디렉토리별 README는 "역할/입력/출력/수정 포인트"를 반드시 포함합니다.
- 새 디렉토리 추가 시 해당 디렉토리에 README를 즉시 추가합니다.
- 기능 PR에는 "어느 디렉토리 책임을 변경했는지"를 명시합니다.
